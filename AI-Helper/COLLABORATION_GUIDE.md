# LIMS 系统协作指南

本文档旨在明确 LIMS (实验室信息管理系统) 项目的开发约定和规范，以确保团队成员 (您与我，AI 助手) 之间的协作顺畅高效。

## 1. 项目目标

- **第一阶段**: 开发一个可简易运行的 LIMS 系统核心功能。我们优先完成业务功能，可以暂时不考虑加密等安全问题，但代码架构需要具备良好的扩展性，以便于未来增加安全性、性能优化等功能，而无需大规模重构。
- **第二阶段**: 在第一阶段的基础上，根据后续讨论的新需求，迭代增加新功能。

## 4. 代码风格

为了保持代码的一致性和可读性，我们约定如下代码风格：

- **C++ (Server)**:
  - **推荐风格**: [Google C++ Style Guide](https://google.github.io/styleguide/cppguide.html)
  - **工具**: 建议使用 `clang-format` 进行自动化格式。
  - **核心原则**:
    - **使用 4 个空格进行缩进。**
    - 类名使用 `PascalCase`，函数和变量名使用 `camelCase`。
    - 在合适的地方添加注释，特别是对于复杂的业务逻辑。
- **Python (Client)**:
  - **强制风格**: [PEP 8 -- Style Guide for Python Code](https://www.python.org/dev/peps/pep-0008/)
  - **工具**:
    - **格式化**: 强烈建议使用 `black` 进行代码自动格式化。
    - **Linter**: 使用 `flake8` 或 `ruff` 来检查代码中的潜在问题。
  - **核心原则**:
    - 使用 4 个空格进行缩进。
    - 遵循 `black` 的默认格式。
